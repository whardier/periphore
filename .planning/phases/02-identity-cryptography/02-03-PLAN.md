<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IPC client → IpcResponse::Identicon | Fingerprint hex + identicon string over Unix socket (0600); same-user access only |
| IPC client → IpcResponse::WordPhrase | Word list over Unix socket; words are public-key-derived, not secret |
| TOML config file → IdentityConfig | show_identicon bool parsed from user-authored file; no env var override supported |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-12 | Information Disclosure | IpcResponse::Identicon over IPC | accept (LOW) | Identicon encodes the public key fingerprint — a public value by design. IPC socket remains 0600 (same user only). No private key material is exposed. ASVS V8.2 (sensitive data logging): identicon is not logged; only fingerprint_hex in GetStatus/tracing is logged, which is the public fingerprint. |
| T-02-13 | Tampering | send_ok() wildcard arm after GetIdenticon/GetWordPhrase removal | mitigate (LOW) | After removing the two stubs from send_ok(), the exhaustiveness check remains satisfied by the `_ => {}` wildcard. Compiler enforces no new IpcCommand variants can silently reach the wildcard without review. Document the removed variants in the comment block in send_ok(). |
| T-02-14 | Spoofing | IpcResponse deserialization of unknown `type` tag | accept (LOW) | serde tag-based enum rejects unknown type strings with a deserialization error. The IPC client (periphore CLI, Phase 5) will receive an error response — acceptable behavior. ASVS V5.3: structured data validated at deserialization. |
| T-02-15 | Information Disclosure | identity.show_identicon env var confusion | accept (LOW) | The field `show_identicon` cannot be set via `PERIPHORE_IDENTITY_SHOW_IDENTICON` due to Figment split("_") producing wrong key path. This is documented in a code comment in IdentityConfig. Users must use TOML. No security impact — failing silently to disable the identicon is not a security risk (the identicon is a public fingerprint representation). |
</threat_model>
