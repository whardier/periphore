**Test 1: `list_cap_eviction` (D-09)**
- Create a `DiscoveredPeerList::new()`.
- Insert 64 entries with `upsert("host-{i}", 7888 + i, DiscoverySource::Mdns, None)` for i in 0..64.
- Verify `list.len() == 64`.
- Insert one more: `upsert("host-overflow", 9999, DiscoverySource::Mdns, None)`.
- Verify `list.len() == 64` (cap maintained).
- Verify the snapshot does NOT contain `host-0` (oldest was evicted).
- Verify the snapshot DOES contain `host-overflow` (newest was inserted).

**Test 2: `gc_removes_expired` (D-08)**

- Insert an entry.
- Verify `list.len() == 1`.

- Verify `list.len() == 1`.
- NOTE: We cannot fast-forward `Instant` in tests without mocking. Instead, test that gc() does NOT remove fresh entries and that the gc function exists. For the actual TTL behavior, add a comment noting that the 5-minute TTL is tested via the constant value check.

**Test 6: `ssh_probe_against_test_listener` (NET-02-SSH)**
This test is more complex -- it starts a minimal TCP listener that speaks the Periphore Hello/HelloAck protocol, then runs the probe against it.

```rust
use tokio::net::TcpListener;
use tokio::sync::mpsc;
use periphore_identity::IdentityStore;
use periphore_net::{codec, PROTOCOL_VERSION};
use periphore_protocol::PeerMessage;
use futures_util::{SinkExt as _, StreamExt as _};

const SEED_A: [u8; 32] = [0u8; 32];
const SEED_B: [u8; 32] = [1u8; 32];

fn make_identity(seed: [u8; 32]) -> (IdentityStore, tempfile::TempDir) {
    let dir = tempfile::tempdir().unwrap();
    let path = dir.path().join("key");
    std::fs::write(&path, seed).unwrap();
    let store = IdentityStore::load_or_create(&path).unwrap();
    (store, dir)
}
```

- Create two identities: `identity_a` (prober) with SEED_A, `identity_b` (remote daemon) with SEED_B.
- Start a TcpListener on port 0 (OS-assigned) that accepts one connection, reads Hello, and responds with HelloAck (accepted: true, using identity_b's fingerprint and public key).
- Get the assigned port.
- Create `DiscoveredPeerList`, `mpsc::channel`, and `CancellationToken`.
- Instead of calling `ssh_probe_loop` directly (it's `pub(crate)`), test by constructing a `DiscoveryService` and calling `start()` with `ssh_probe_enabled = true` and `ssh_probe_ports = vec![assigned_port]`.
- Wait for up to 5 seconds for a `DiscoveryEvent::PeerDiscovered` on the channel.
- Cancel the service.
- Verify the event has `hostname = "127.0.0.1"`, `port = assigned_port`, `source = DiscoverySource::SshProbe`.

NOTE: If `ssh_probe_loop` is not directly testable from integration tests (it's `pub(crate)`), test via the `DiscoveryService` public API instead. The test listener must respond with a valid HelloAck. If this approach has issues with the probe's 30-second interval, adjust: the probe runs immediately on start before waiting, so the first sweep should happen within the first few seconds.

Actually, re-reading the probe.rs design: the probe loop does NOT sleep before the first sweep -- it probes all ports first, THEN sleeps. So the first discovery event should arrive within ~300ms (connect timeout + handshake timeout for one port). This is testable.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| daemon select! -> discovery channel | DiscoveryEvent flows from discovery tasks to daemon; internal channel, no external trust boundary |
| IPC socket -> daemon | Local CLI sends GetDiscoveredPeers; socket is 0600 (owner-only) |
| SSH probe -> test listener | Integration test listener simulates a Periphore daemon for probe validation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-10 | Spoofing | main.rs (discovery event handling) | accept | Discovery events come from internal mpsc channel (not external network). Spoofing requires compromising the process itself. |
| T-7-11 | Denial of Service | main.rs (discovery_event_rx.recv()) | mitigate | Channel buffer is 64 (bounded); if discovery tasks produce events faster than daemon processes them, back-pressure applies. No unbounded growth. |
| T-7-12 | Repudiation | integration.rs (test listener) | accept | Test-only code; no production trust boundary. Test listener uses deterministic seeds for reproducibility. |
</threat_model>
